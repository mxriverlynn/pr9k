## Issue #10 — ralph-tui: UI keyboard shortcuts and shortcut bar

Implemented the keyboard shortcut state machine in `ralph-tui/internal/ui/ui.go`:

- `StepAction` type with constants: `ActionRetry`, `ActionContinue`, `ActionQuit`
- `Mode` type with constants: `ModeNormal`, `ModeError`, `ModeQuitConfirm`
- Shortcut text constants: `NormalShortcuts`, `ErrorShortcuts`, `QuitConfirmPrompt`
- `KeyHandler` struct with `ShortcutLine` string (Glyph reads via pointer), `Actions` channel, and `cancel` func
- `NewKeyHandler(cancel func(), actions chan StepAction)` constructor
- `Handle(key string)` method dispatches keypresses to mode-specific handlers
- `SetMode(mode Mode)` switches modes from the orchestration goroutine (e.g., on step failure)
- Normal mode: `n` calls cancel (subprocess skip), `q` enters quit-confirm
- Error mode: `c` sends ActionContinue, `r` sends ActionRetry, `q` enters quit-confirm
- Quit-confirm mode: `y` sends ActionQuit, `n` restores previous mode, other keys ignored

Tests in `ralph-tui/internal/ui/ui_test.go` cover all acceptance criteria (10 test functions).

### Acceptance criteria status
- [x] Shortcut bar displays correct keys for normal mode
- [x] Shortcut bar updates to error-specific keys when a step fails
- [x] `n` during a running step triggers subprocess cancellation
- [x] `q` shows quit confirmation prompt
- [x] `y` during quit confirmation exits the TUI
- [x] `n` during quit confirmation restores normal shortcut bar
- [x] `c` in error state sends `ActionContinue` on the channel
- [x] `r` in error state sends `ActionRetry` on the channel
- [x] Other keys during quit confirmation are ignored

## Issue #3 — ralph-tui: Prompt building with variable prepending

Implemented `BuildPrompt(projectDir string, step Step, issueID string, startingSHA string) (string, error)` in `ralph-tui/internal/steps/steps.go`.

- Reads prompt file from `projectDir/prompts/<step.PromptFile>`
- When `step.PrependVars` is true, prepends `ISSUENUMBER=<id>\nSTARTINGSHA=<sha>\n` (real newlines) before file content
- When `step.PrependVars` is false, returns raw file content unchanged
- Returns wrapped error if prompt file cannot be read

Added 5 tests to `steps_test.go`:
- `TestBuildPrompt_PrependVarsTrue` — verifies prepended output matches expected string exactly
- `TestBuildPrompt_PrependVarsFalse` — verifies output equals raw file content
- `TestBuildPrompt_FileNotFound` — verifies error returned for missing prompt file
- `TestBuildPrompt_RealNewlines` — verifies real `\n` chars present, no literal `\n` sequences
- `TestBuildPrompt_CorrectInterpolation` — verifies issue ID and SHA are correctly embedded

### Acceptance criteria status
- [x] Reads the prompt file from `projectDir/prompts/<promptFile>`
- [x] When `prependVars` is true, output starts with `ISSUENUMBER=<id>\nSTARTINGSHA=<sha>\n` followed by file content
- [x] When `prependVars` is false, output equals raw file content exactly
- [x] Returns an error if the prompt file cannot be read
- [x] Uses real newlines (not literal `\n` characters)
